# /qa — Web アプリケーション QA テスト & 修正

> Source: gstack `/qa` (Garry Tan) のコンセプトを ECC に適合

## 目的

Webアプリケーションを体系的にテストし、発見したバグを修正してコミットする。
`e2e-runner` エージェントがテスト自動化に特化するのに対し、`/qa` は手動検証に近いビジュアル確認 + 修正ループを提供する。

## Step 1: テスト対象の特定

1. dev server が起動しているか確認:
   ```bash
   curl -s -o /dev/null -w "%{http_code}" http://localhost:3000 2>/dev/null || echo "NOT_RUNNING"
   ```
2. 起動していなければユーザーに確認（URL、ポート番号）
3. テスト対象のページ/機能を特定（ユーザー指定 or 変更差分から推定）

## Step 2: テスト範囲の決定

| Tier | 対象 | 所要時間 |
|------|------|---------|
| **Quick** | Critical + High のみ（主要フロー、認証、決済） | 5-10分 |
| **Standard** | + Medium（フォームバリデーション、エラーハンドリング、レスポンシブ） | 15-30分 |
| **Exhaustive** | + Cosmetic（タイポ、アライメント、ホバー状態、アクセシビリティ） | 30-60分 |

ユーザーが指定しなければ Standard をデフォルトにする。

## Step 3: テスト実行

`webapp-testing` スキルの Playwright を使い、以下のチェックリストを体系的に検証する:

### Critical（全 Tier）
- [ ] ページが正常にロードされる（200応答、白画面なし）
- [ ] 主要ユーザーフローが完走する（登録→ログイン→主機能→ログアウト）
- [ ] 認証が正しく動作する（未認証リダイレクト、セッション切れ）
- [ ] 決済フローにエラーがない（該当する場合）

### High（全 Tier）
- [ ] フォーム送信が正常に動作する（成功・バリデーションエラー）
- [ ] API エラー時にユーザーフレンドリーなメッセージが表示される
- [ ] ブラウザの戻る/進むが壊れない
- [ ] コンソールに重大なエラーがない

### Medium（Standard 以上）
- [ ] モバイルビューで崩れがない
- [ ] 空状態（データなし）の表示が適切
- [ ] ローディング状態が表示される
- [ ] 入力バリデーションが正しく動作する（境界値、特殊文字）

### Cosmetic（Exhaustive のみ）
- [ ] テキストの誤字脱字がない
- [ ] 要素のアライメントが揃っている
- [ ] ホバー/フォーカス状態が適切
- [ ] ダークモード対応（該当する場合）

各チェック項目について:
1. Playwright でスクリーンショットを取得
2. コンソールログを確認
3. 問題があれば Issue として記録

## Step 4: バグ修正ループ

発見した Issue を重要度順にソートし、1件ずつ修正する:

```
for each issue (Critical → High → Medium → Cosmetic):
  1. 原因を特定（ソースコード読み込み）
  2. 修正を実装（最小差分）
  3. 修正後に同じチェックを再実行（リグレッション確認）
  4. 問題なければ次の Issue へ
```

## Step 5: レポート出力

```markdown
## QA Report — {プロジェクト名}

**Tier**: Standard
**URL**: http://localhost:3000
**日時**: YYYY-MM-DD HH:MM

### Health Score
Before: X/10 → After: Y/10

### 発見した Issue (N件)

| # | 重要度 | 内容 | ステータス |
|---|--------|------|----------|
| 1 | Critical | ... | Fixed |
| 2 | High | ... | Fixed |
| 3 | Medium | ... | Known (次スプリント) |

### 修正内容
- {ファイル}: {変更概要}

### 未修正・既知の問題
- {理由付きで列挙}

### Ship Readiness
**READY** / **NOT READY** (理由)
```

## e2e-runner との棲み分け

| 観点 | /qa | e2e-runner |
|------|-----|-----------|
| 目的 | ビジュアル確認 + バグ修正 | 自動テストスイート構築 |
| 出力 | レポート + 修正コミット | テストファイル (.spec.ts) |
| タイミング | フィーチャー完成後の検証 | CI/CD に組み込む継続テスト |
| 人間で言うと | QAエンジニアの手動テスト | テスト自動化エンジニア |
