# Session Report — 作業記録作成スキル

プロジェクトの作業記録を `.company/subsidiaries/{子会社}/engineering/` 配下に保存する。
git diff と会話履歴から自動的に内容を構成し、既存レポートがあれば追記する。

## ワークフロー

### Step 1: プロジェクト特定

現在の作業対象プロジェクトを特定する。以下の順で試行:

1. 会話中で明示的に言及されたプロジェクト名
2. 直近で編集したファイルのパスから推定
3. `~/.company/pm/CLAUDE.md` のプロジェクト一覧と照合

特定したら、対応する子会社を `~/.company/CLAUDE.md` の子会社テーブルから取得する。

### Step 2: 変更情報の収集

プロジェクトディレクトリで以下を実行:

```bash
# 変更ファイル一覧
git diff --stat

# 直近のコミット（今日分）
git log --oneline --since="midnight"

# ステージされていない変更も含む
git status --short
```

会話履歴からも以下を抽出:
- 実施した作業の概要
- 遭遇したエラーと解決方法
- 意思決定の記録
- 未完了タスク

### Step 3: レポート作成・更新

**保存先**: `~/.company/subsidiaries/{子会社}/engineering/YYYY-MM-DD-session-report.md`

同日のレポートが既に存在する場合は、新しいセッションセクションを追記する（上書きしない）。

#### 新規作成時のテンプレート

```markdown
# YYYY-MM-DD セッションレポート: {プロダクト名} ({子会社名})

## 実施内容
1. {作業1の概要}
2. {作業2の概要}
...

## 変更ファイル
- {ファイルパス}（{変更概要}）
- ...

## 次にやること
- [ ] {未完了タスク1}
- [ ] {未完了タスク2}
- ...
```

#### 追記時のテンプレート

既存ファイルの末尾に以下を追記:

```markdown

## セッションN: {セッションの主題}（MM/DD HH時頃）

### 実施内容
1. ...

### 変更ファイル
- ...

### 次にやること（更新）
- [ ] ...
```

### Step 4: タスクファイル更新（必須）

レポート作成後、**必ず** `.company/secretary/todos/YYYY-MM-DD.md` を更新する:

1. レポートの「実施内容」に対応するタスクに `[x]` を付ける
2. レポートの「次にやること」から新規タスクを追加する（T-NNN 番号付き）
3. ★手動 タグが必要な項目にはタグを付ける

このステップは省略禁止。レポートとタスクファイルの整合性を保つ。

### Step 5: 関連ファイルの更新

レポート作成後、以下も必要に応じて更新:

1. **子会社 engineering/CLAUDE.md**: ドキュメント一覧にレポートを追加
2. **スプリント**: `current-sprint.md` の該当タスクにチェックを入れる（完了した場合）

### Step 6: 確認

作成・更新したファイルパスを報告する。内容のプレビューは不要（ユーザーの feedback メモリ: diff やコードを見せず「更新した・ファイルパス」だけ報告する）。

## 注意事項

- セッション中に複数プロジェクトを跨いだ場合は、プロジェクトごとに別レポートを作成する
- 「次にやること」には ★手動 タグを付けて、人間の手作業が必要な項目を明示する
- git diff が取れない場合（非 git ディレクトリなど）は会話履歴のみから構成する
- 機密情報（API キー、パスワード等）をレポートに含めない
